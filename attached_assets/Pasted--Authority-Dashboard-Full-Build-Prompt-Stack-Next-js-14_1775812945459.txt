## SECTION A — AGENT GROUND RULES

1. *App Router only.* Authority routes live under app/authority/. No Pages Router.
2. **Every authority query MUST filter by the logged-in user's department_id.** Never fetch all reports — always scope to the department.
3. **'use client'** is required on every component using hooks, events, or browser APIs.
4. *Mapbox is client-only.* Always use dynamic(() => import(...), { ssr: false }).
5. *Do NOT use the adminClient in authority pages or components.* Authority users use the standard Supabase server/client with their own session and RLS policies.
6. **Status updates must write to both reports and activity_logs** in a single server action or route call.
7. *Never display raw UUIDs* to authority users. Always show report_number (format: NS-2025-000001).
8. *Citizen phone numbers must be masked* in the authority view: show 017*****89 pattern.
9. *Never invent Supabase methods.* Only: supabase.from(), supabase.storage, supabase.auth, supabase.rpc().
10. **All data mutations (status update, etc.) go through app/api/authority/*/route.ts** — not direct Supabase calls from client components.

## SECTION B — AUTHORITY DATABASE SETUP

Run this SQL in Supabase SQL Editor AFTER Part 1 schema is applied:

-- AUTHORITY USERS TABLE
CREATE TABLE authority_users (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  department_id UUID NOT NULL REFERENCES departments(id),
  username TEXT UNIQUE NOT NULL,
  full_name TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('viewer','manager','admin')),
  is_active BOOLEAN DEFAULT TRUE,
  auth_user_id UUID REFERENCES auth.users(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS FOR AUTHORITY USERS
ALTER TABLE authority_users ENABLE ROW LEVEL SECURITY;
CREATE POLICY "authority_users_select_own" ON authority_users
  FOR SELECT USING (auth_user_id = auth.uid());

-- AUTHORITY CAN READ REPORTS FOR THEIR DEPARTMENT
CREATE POLICY "reports_authority_dept" ON reports
  FOR SELECT USING (
    department_id IN (
      SELECT department_id FROM authority_users WHERE auth_user_id = auth.uid() AND is_active = true
    )
  );

-- AUTHORITY CAN UPDATE REPORT STATUS
CREATE POLICY "reports_authority_update" ON reports
  FOR UPDATE USING (
    department_id IN (
      SELECT department_id FROM authority_users WHERE auth_user_id = auth.uid() AND is_active = true
    )
  );

-- AUTHORITY CAN READ IMAGES FOR THEIR REPORTS
CREATE POLICY "images_authority_dept" ON report_images
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM reports r
      JOIN authority_users au ON au.department_id = r.department_id
      WHERE r.id = report_images.report_id AND au.auth_user_id = auth.uid()
    )
  );

-- AUTHORITY CAN READ AI ANALYSIS FOR THEIR REPORTS
CREATE POLICY "ai_analysis_authority" ON ai_analysis
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM reports r
      JOIN authority_users au ON au.department_id = r.department_id
      WHERE r.id = ai_analysis.report_id AND au.auth_user_id = auth.uid()
    )
  );

-- AUTHORITY CAN READ ACTIVITY LOGS
CREATE POLICY "activity_logs_authority" ON activity_logs
  FOR SELECT USING (
    EXISTS (
      SELECT 1 FROM reports r
      JOIN authority_users au ON au.department_id = r.department_id
      WHERE r.id = activity_logs.report_id AND au.auth_user_id = auth.uid()
    )
  );

-- AUTHORITY CAN INSERT ACTIVITY LOGS (for status changes)
CREATE POLICY "activity_logs_authority_insert" ON activity_logs
  FOR INSERT WITH CHECK (actor_type = 'authority');

## SECTION M — ERROR HANDLING & DEBUGGING GUIDE

### M1 — Authority Auth Errors

| Error | Cause | Fix |
|---|---|---|
| Redirect loop to /authority/login | Session exists but authority_users record missing | Run the INSERT SQL from Section B Step 2 — the auth user must have a matching row in authority_users |
| authorityUser is null even after INSERT | auth_user_id UUID mismatch | In Supabase Auth dashboard, copy the exact UUID of the user. The UUID in the INSERT must match exactly. |
| Can't see any reports after login | RLS policy misconfigured | Run debug query below |
| "JWT expired" error | Session expired | Call supabase.auth.refreshSession() or redirect to login |

*Debug authority session:*
// In layout.tsx, temporarily add:
console.log('Auth session user id:', session?.user.id)
console.log('Authority user found:', JSON.stringify(authorityUser))

*Debug RLS for authority:*

SELECT au.id, au.username, au.department_id, au.auth_user_id, d.slug
FROM authority_users au
JOIN departments d ON d.id = au.department_id;

SELECT * FROM reports LIMIT 5;

### M2 — Report Query Errors

| Symptom | Cause | Fix |
|---|---|---|
| Returns 0 reports even though reports exist | Reports have department_id = null | Reports submitted before the routing fix will have null dept. Run: UPDATE reports SET department_id = (SELECT id FROM departments WHERE slug = 'dncc') WHERE department_id IS NULL |
| count is null in the count query | Forgot { count: 'exact', head: true } options | Add both options to the .select() call |
| Reports from other departments appearing | Missing .eq('department_id', deptId) filter | Always add this filter to EVERY query in authority routes |
| Stale data after status update | Component not re-fetching | Call router.refresh() after a successful mutation (this re-runs server component fetches) |

### M3 — Status Update Errors

| Error | Cause | Fix |
|---|---|---|
| 403 from /api/authority/update-status | Report doesn't belong to this department | The route handler must verify report.department_id === authorityUser.department_id before updating |
| Activity log INSERT fails | Missing RLS policy for authority inserts | Run: CREATE POLICY "activity_logs_authority_insert" ON activity_logs FOR INSERT WITH CHECK (actor_type = 'authority') |
| Status change doesn't reflect on page | router.refresh() not called | After a successful fetch response, call router.refresh() to re-run server component data fetching |

*Debug status update route:*
// In update-status/route.ts, add:
console.log('[Status Update]', { reportId, newStatus, authorityUserId, reportDeptId, authorityDeptId })
// This will show in your terminal and help trace 403 issues

| Symptom | Fix |
|---|---|
| ReferenceError: window is not defined | Add 'use client' to AnalyticsCharts.tsx |

### M5 — Mapbox (Report Detail Map) Errors

| Symptom | Fix |
|---|---|
| Map blank / grey | mapboxgl.accessToken not set before map init. Set it in useEffect before new mapboxgl.Map() |
| Cannot read properties of undefined (reading 'accessToken') | mapbox-gl imported in a server context — ensure component has 'use client' and is imported with dynamic(..., { ssr: false }) |
| Marker at wrong location | Mapbox uses [longitude, latitude] order — opposite of Google Maps. Double-check the order. |

| EXIF metadata all null | Depends on the image. Most modern smartphone photos have EXIF. Screenshots and downloaded images usually don't. This is expected behavior — display "EXIF data not available" gracefully. |

### M7 — Common Next.js Mistakes in This Module

| Mistake | Symptom | Fix |
|---|---|---|
| useAuthority() used in server component | Build error | Authority context is client-only. Server components receive authorityUser via props from the layout. |
| cookies() called in a utility function, not a component | Error: called outside request | Move the createClient() call into the Server Component body, not into a utility |
| Forget to add notFound() when report not found | Blank page or hydration error | Always add if (!report) notFound() after fetching a single report by ID |
| Forgetting router.refresh() after mutation | UI shows stale status after update | router.refresh() is the correct way to re-fetch server component data without a full page reload |
